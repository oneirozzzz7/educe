# Educe 能力边界重设计：框架作为资产容器

> 第五公理的彻底落地。来自 2026-06-24 Opus 4.8 × 5 轮架构讨论。

---

## 一、核心命题

**框架不持有任何关于"什么是对的"的知识。它只持有"关于这个用户的事实"，并保证在不可逆的那一刻，把这些事实摆到决策者面前。**

模型越强，同样的事实产出越好的决策——框架不改一行代码，能力却升级了。

---

## 二、三方角色定义

| 角色 | 职责 | 不做什么 |
|------|------|----------|
| **框架** | 持有用户资产 + 召回相关信息 + 记录反馈 + 不可逆暂停机制 | 不做任何判断、不仲裁、不分级、不推荐 |
| **模型** | 所有决策：做什么、怎么做、危不危险、要不要问用户 | 不存储跨会话状态（失忆的） |
| **用户** | 声明边界 + 确认不可逆 + 提供反馈 + 授权泛化 | 不需要频繁被打扰（成本应随经验下降） |

**权力流转规则**：
- 框架 → 模型：通过信息注入增强模型决策质量
- 模型 → 用户：模型自己判断"我不确定"时上交用户
- 用户 → 框架：用户反馈沉淀为资产，下次模型能看到
- 框架永远不决策，只在不可逆前暂停

---

## 三、框架的三个组件

### 3.1 AssetStore（用户资产容器）

四类资产，全部是**可被模型阅读的自然语言/结构化事实**，不是可执行逻辑：

| 资产类型 | 内容示例 | 积累方式 |
|----------|----------|----------|
| **先例 (Precedent)** | "在删除 build/ 时用户批准了，理由：构建产物可重建" | Ledger 自动沉淀 |
| **偏好 (Preference)** | "用户倾向先 plan 再执行" | 多次先例由模型归纳，用户确认 |
| **知识 (Knowledge)** | "本项目用 pnpm；prod 在 10.0.1.5" | 用户显式提供 / 模型探索后确认 |
| **边界 (Boundary)** | "不要 force push main；prod 数据库只读" | 用户显式声明 |

**设计原则**：
- 资产是数据不是代码，模型直接阅读
- 资产有时间戳，支持衰减和废止
- 资产之间可以矛盾（矛盾本身是信息，交模型处理）
- 用户可以随时查看、修改、删除自己的资产

### 3.2 ContextAssembler（信息召回层）

**职责**：当模型要执行动作时，从 AssetStore 中召回相关资产，注入模型上下文。

```
模型要执行: rm -rf build/
召回结果注入:
  - [边界] 用户声明: "prod 目录不可删除" (2天前)
  - [先例] 上次删除 build/ 用户批准, 理由"构建产物" (5天前)
  - [知识] build/ 是 next.js 构建产出, 可通过 pnpm build 重建
```

模型看到这些信息后，自己决定：这是 build/ 不是 prod/，有先例，直接执行。

**关键约束**：
- ContextAssembler 不判断"该不该执行"，只判断"哪些资产跟当前动作相关"
- 召回策略可以简单开始（关键词匹配），随时间演进（语义检索）
- 召回结果永远是信息，不是指令

### 3.3 Ledger（记录器）

**职责**：忠实记录每次交互的决策和反馈。

记录内容：
- 模型做了什么决策（action + context）
- 用户的反应（批准/拒绝/撤销/无反应）
- 决策的结果（成功/失败/部分成功）

**沉淀规则**：
- Ledger 只记录，不分析
- 从 Ledger 到 AssetStore 的沉淀由模型归纳（"看这几条记录，有什么规律？"）
- 用户确认后才正式成为资产

## 四、唯一硬编码：不可逆暂停机制

### 什么是硬编码的

```python
# 框架唯一的硬逻辑：不可逆动作前暂停
if action.is_irreversible:
    relevant_assets = assembler.recall(action)
    present_to_decision_maker(relevant_assets)
    await explicit_clearance()  # 用户或模型显式放行
```

### 什么不是硬编码的

- "什么算不可逆" — 部分是物理事实（删除/支付/对外发送），部分来自用户边界声明
- "危不危险" — 完全交给模型判断，框架不参与
- "要不要问用户" — 模型自己决定，基于看到的资产和自身确定性
- "先例够不够强" — 模型自己判断

### is_irreversible 的定义

分两层：
1. **系统级不可逆**（物理事实，极少）：文件删除、对外网络请求、进程 kill
2. **用户声明的不可逆**（边界资产）：用户说"prod 对我来说不可逆"就是不可逆

两层的并集触发暂停。这个并集会随用户资产增长而变化，但"暂停机制本身"不变。

## 五、与现有模块的关系

### 废弃

| 模块/机制 | 原因 |
|-----------|------|
| `binary_exts` 硬编码列表 | 框架替模型判断"什么能读" |
| `shell_taxonomy.yaml` 静态分级 | 框架替模型判断"什么危险" |
| Evidence-Gated Nudge 的收敛判断 | 框架替模型判断"该停了" |
| CompositeSkill 自动注入 | 框架替模型选 skill |
| Arbiter 仲裁逻辑 | 框架做裁判 |

### 保留但改性质

| 模块 | 原来 | 改为 |
|------|------|------|
| reflex_router | 框架硬编码 pattern 直接执行 | 回放模型曾经的成功决策（资产驱动，非硬编码） |
| governance.py | 决策拦截 + 信息提供混合 | 纯信息层（呈现 Ledger/Asset 给前端） |
| confirm 机制 | 框架决定什么需要 confirm | 仅为不可逆暂停服务 + 模型主动请求 |
| memory_store | 只存知识 | 扩展为 AssetStore 的底层存储 |

### 新建

| 模块 | 职责 | 预估规模 |
|------|------|----------|
| `asset_store.py` | 四类资产的 CRUD + 检索 | ~300行 |
| `context_assembler.py` | 相关资产召回 + 上下文注入 | ~200行 |
| `ledger.py` | 决策+反馈记录 | ~150行 |
| `irreversibility.py` | 不可逆判定（系统级 + 用户边界） | ~80行 |

## 六、Confirm 成本下降机制

用户不应该被反复打扰。Confirm 成本随经验下降的路径：

### 下降路径 1：先例消解

模型看到先例"用户上次批准了类似操作" → 模型自己决定不问，直接执行。

**注意**：这不是框架自动消解 confirm，是模型读到先例后自己的判断。模型更强 → 更精准地判断"这次和上次够像" → 更少打扰用户。

### 下降路径 2：边界排除

用户声明边界覆盖了该场景 → 模型看到边界明确允许 → 不问。

### 下降路径 3：偏好学习

模型归纳出偏好"这个用户对构建产物的删除一向宽松" → 整体减少该类 confirm。

### 保底

模型不确定时，仍然问用户。但随着资产积累，"不确定"的范围会收缩。

## 七、框架在模型升级世界中的价值

框架的价值锁定在**模型永远拿不到的东西**上：

1. **跨会话记忆** — 模型每次失忆，框架记得这个用户是谁
2. **用户私有事实** — 模型训练数据里没有你的环境
3. **反馈闭环捕获** — 模型不知道"上次我做了X用户撤销了"

这三样随使用时长 O(t) 增长，与模型能力正交。

**模型升级时的表现**：
- 弱模型 + 多资产 = 能做出合理决策（资产弥补推理不足）
- 强模型 + 多资产 = 能做出精准决策（资产+推理叠加）
- 强模型 + 零资产 = 通用但不了解你（这就是裸 ChatGPT）

框架越薄、资产越厚 → 用户锁定越强、体验越好。

## 八、与五条公理的对齐

| 公理 | 本设计如何体现 |
|------|---------------|
| **代谢** | Ledger 记录 → AssetStore 沉淀 → ContextAssembler 喂回 = 完整闭环 |
| **三层下沉** | 先例从"每次问模型"下沉为"模型看一眼就知道" = 认知沉积 |
| **收敛优先** | 资产在特定任务族上积累密度 = 自然收敛 |
| **环境即身体** | AssetStore 就是 seed 在这个环境里长出的"记忆体" |
| **机制与认知分离** | 框架 = 纯机制（存/取/暂停）；认知 = 全在资产里（来自用户/模型/探测） |

## 八-B、环境观测层 + 产物交付（第 7-8 轮补充）

### 核心原则

**工具自报的是意图，Observer 实测的是事实。**

框架不信任 `write_file` 返回的 `success=True`。框架自己去 stat 文件，验证环境变化。

### 环境观测器 (EnvironmentObserver)

- write_file 执行后：框架独立 stat 文件，确认存在+大小+hash
- shell 执行前后：对工作目录做浅层快照(mtime)，diff 发现新增/修改/删除
- 所有 `created`/`modified` 的文件自动登记为 Artifact

### Artifact = 可交付产物

```
Artifact {
  artifact_id, path, filename, mime, size, hash,
  created_by (哪个 tool_call), preview_kind (code|image|text|binary),
  download_url → /api/artifacts/{id}
}
```

- Artifact 由 observer 创建（不是模型声称的）
- 文件不存在 → 没有 artifact → 幻觉自然不被交付
- 前端收到 `artifact_created` 事件 → 渲染预览/下载卡片

### 任务完成的硬指标

**产物未交付 = 任务未完成。** 这是事实不是判断：
- 事实："本次会话产生了 3 个文件，0 个被展示给用户"
- 不需要理解"用户到底要什么"

runtime_facts 注入：
```
files_produced_this_session:
  - hello.py (code, 22B, by write_file)
  - fibonacci.py (code, 294B, by write_file)
delivery_status: 2 artifacts registered, available via /api/artifacts
```

### API 端点

- `GET /api/artifacts` — 列出所有产物
- `GET /api/artifacts/{id}` — 下载/预览单个产物

## 九、重构阶段计划

### Phase 0：埋点 + 审计（3天）

**做什么**：在 orchestrator 现有的每个决策点记录 `{who_decided, what, context, outcome}`。

**产出**：一份"框架当前隐式做了多少决策"的审计报告。

**验证**：跑一天真实使用，能回答"今天框架替模型/用户做了 N 个决策"。

### Phase 1：AssetStore + Ledger（1周）

**做什么**：
1. 新建 `asset_store.py` — 四类资产 CRUD
2. 新建 `ledger.py` — 决策+反馈记录
3. 现有 memory_store 的 fact/scar/convention 迁移为 Asset

**验证**：手动写入几条资产，确认可以按类型/关键词检索。

### Phase 2：ContextAssembler 接入主循环（1周）

**做什么**：
1. 新建 `context_assembler.py` — 根据当前 action 召回相关资产
2. orchestrator 在每次 LLM 调用前，注入召回的资产
3. **不改变任何决策逻辑** — 模型多看到了信息，但框架行为不变

**验证**：观察日志，确认资产被正确注入到 prompt 中。模型的回答质量应有可观测的改善。

### Phase 3：拆除硬编码判断（2周）

**做什么**：
1. 删除 `binary_exts` — read_file 只做内容检测+元信息（已完成）
2. 删除 `shell_taxonomy.yaml` 的 block/confirm 逻辑 — 改为不可逆暂停
3. Evidence-Gated Nudge 降级为信息注入（"你已经循环5次了" → 注入上下文，不强制收敛）
4. CompositeSkill 从自动注入改为能力清单暴露

**验证**：
- 正向：原来被 binary_exts 拦截的 .json/.md 能正常读取
- 正向：原来被 taxonomy 误判的 bash/echo 不再弹 confirm
- 负向：真正不可逆的操作（rm -rf /）仍然暂停
- 回归：跑 benchmark 确认没有能力退化

### Phase 4：Confirm 成本下降验证（2周）

**做什么**：
1. Ledger 开始自动记录用户对 confirm 的回答
2. 资产沉淀：连续 N 次批准同类操作 → 模型归纳为先例
3. 模型下次看到先例 → 自己决定不问

**验证**（The Descent Curve 的雏形）：
- 选一个操作类型，重复执行 10 次
- 前 2-3 次应该有 confirm，后面应该逐渐消失
- 画出 confirm 频率曲线，应单调下降

### Phase 5：前端可视化（1周）

**做什么**：
- 资产管理界面（查看/编辑/删除）
- 决策日志（谁做了什么决策，依据是什么）
- 先例的"授权链"可视化

**验证**：用户能在界面上看到"这个操作没问你，因为模型看到了先例#42"。

## 十、风险和防护

| 风险 | 防护 |
|------|------|
| ContextAssembler 召回太多噪音 | 限制注入条数（top-5），逐步调优 |
| 资产积累太慢，短期体验无改善 | Phase 3 的硬编码拆除本身就解决了误判 bug |
| 模型升级后先例变得多余 | 无害——模型自己会忽略不需要的先例 |
| 不可逆判定太宽/太窄 | 用户可随时声明/撤销边界 |
| 先例被恶意利用（注入攻击） | 先例来源必须是 Ledger（用户真实交互），不接受外部输入 |

## 十一、成功标准

整个重构成功的判据：

1. **框架代码里搜不到业务判断** — 没有 "if dangerous"、没有扩展名列表、没有分级 yaml
2. **同一操作的 confirm 频率随使用下降** — The Descent Curve 在 confirm 维度成立
3. **模型升级后零修改** — 换一个更强的模型，框架不改代码，体验自动变好
4. **用户能看到且控制自己的资产** — 透明、可审计、可撤销
