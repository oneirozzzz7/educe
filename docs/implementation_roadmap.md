# DeepForge 实施路线（重写版）

## 核心原则

1. **先验证假设再写代码**——不基于猜测设计方案
2. **一次只加一个机制**——验证不冲突再加下一个
3. **测试场景从真实使用中来**——不自己编理想化问题
4. **哲学检验在编码前做**——每个改动先问"帮助还是替代"

---

## Phase 0：验证基础假设（不写代码，只做实验）

### 假设1：弱模型能否从 conversation history 正确理解跨轮意图？

**实验方法**：
不加任何辅助机制，只把 conversation history 传给模型，看它能否正确处理"做完代码→问论文"这类场景。

```
实验步骤：
1. 启动一个干净session
2. 发消息"做一个番茄钟"→模型生成代码
3. 代码在conversation history里
4. 发消息"什么是AI"→看模型是否回复文本（而非修改代码）
5. 发消息"把刚才的番茄钟改成暗色"→看模型是否理解是改代码
6. 发消息"谢谢"→看模型是否正常回复

如果4步全对：弱模型能理解上下文，不需要ContextAnalyzer
如果有错：记录哪步错了，分析为什么错，再决定需要什么辅助
```

**关键**：当前代码里有 `_is_text_task` 和 `_run_modify` 这些硬编码路由，会干扰实验。需要先把它们注释掉，让模型自己走 `_decide` 判断。

### 假设2：知识注入是提升还是噪声？

**实验方法**：
```
实验步骤：
1. 清空知识库
2. 跑10题（第1批）→记录得分A（无知识注入）
3. 人工检查第1批提取的insight质量
4. 跑同样10题（第2批）→记录得分B（有知识注入）
5. 对比A和B
6. 如果B < A：知识注入是噪声，需要改策略
7. 如果B >= A：知识注入有效
8. 跑另外10题（第3批，不同问题）→记录得分C
9. C和A对比：跨问题的知识是否有帮助还是有害
```

### 假设3：用户行为信号能否被准确检测？

**实验方法**：
```
用真实的对话记录（用户截图里的使用模式）：
1. "介绍下京东公司" → 长文本回复
2. "帮我做一个坦克大战游戏呗" → 代码生成
3. "上面提及的论文对自进化有什么参考价值吗" → 文本追问

人工标注每步的预期信号：
  turn 2 相对 turn 1: topic_switch（换话题）✓
  turn 3 相对 turn 2: 这是追问上面提到的论文 ✓

用 detect_user_signal 检测，看是否匹配人工标注
```

### 假设4：激发语的差异到底有多大？

**实验方法**：
```
用3个不同的激发语变体，同一组10题，跑3遍
统计方差——如果3个变体的得分差异小于LLM输出的随机性（同一变体跑3遍的方差），
则激发语演化没有意义，应该停止在这个方向投入。
```

**Phase 0 的产出**：4个假设的验证结果 → 决定哪些机制真正需要做

## Phase 1：修复当前已知 bug（基于 Phase 0 结果）

### 不管假设结论如何，这些 bug 必须修：

**Bug 1：做完代码后问论文→输出代码**
- 根因：`_run_modify` 自动触发
- 修复取决于假设1的结论：
  - 如果弱模型能理解history → 删掉 `_run_modify` 自动触发，让模型自己判断
  - 如果弱模型理解不了 → 设计最小化的辅助（不是ContextAnalyzer全家桶，而是一句话提示）

**Bug 2：文本聊天不保存历史**
- 根因：`task_store.save_from_context` 只在code路径调
- 修复：text路径也保存，且保存response内容
- 这个不依赖任何假设，直接修

**Bug 3：侧栏历史点击不能回看文本回复**
- 根因：save时没有response字段
- 修复：task_store增加response字段

### Phase 1 验收标准

```
必须通过的场景（从用户真实使用中提取）：
- [ ] 介绍京东→做坦克大战→问论文 → 三步都正确路由
- [ ] 上传PDF→解读→追问细节→换话题 → 文件上下文正确管理
- [ ] 连续5轮闲聊 → 侧栏全部显示 + 都能回看
- [ ] 做计算器→改颜色→问天气→改字体 → 代码修改和聊天正确交替
- [ ] Playwright 25/26
```

## Phase 2：知识蒸馏（基于假设2结论）

**只有假设2验证"知识注入有效"时才做这个Phase。**

如果假设2显示知识注入是噪声：
- 停止在知识注入方向投入
- 转向其他方向（比如context管理或prompt优化）

如果假设2显示有效但提升微小（<3%）：
- 做，但不作为核心机制
- 简化实现，不追求完美

如果假设2显示有效且提升显著（>5%）：
- 全力投入
- 精细化提取策略和召回策略

### Phase 2 验收标准

```
- [ ] 越用越强闭环：第2批10题 > 第1批10题（同一批问题）
- [ ] 跨问题知识：第3批不同问题不低于第1批（不引入噪声）
- [ ] 知识质量人工抽查：随机5条insight，4条以上是有价值的
- [ ] 得分不低于Phase 1
```

---

## Phase 3：激发语演化（基于假设4结论）

**只有假设4验证"激发语差异大于随机噪声"时才做。**

如果假设4显示差异不显著：
- 停在当前的涌现式激发语不动
- 把时间投入到其他更有价值的方向

如果差异显著：
- 实现 epsilon-greedy 选择
- 实现基于质量数据的淘汰

### Phase 3 验收标准

```
- [ ] 3个变体的得分差异 > 同一变体3次跑的方差
- [ ] 薄弱领域选择了非默认变体后得分提升
- [ ] 得分不低于Phase 2
```

---

## Phase 4-6：根据前面的结果决定

Phase 4 (UserProfile) / Phase 5 (CredibilityEngine) / Phase 6 (ContextDistillation) 
是否需要做、以什么优先级做，取决于 Phase 0-3 的结果。

**不预先规划太远。** 每个Phase完成后评估：
- 当前最大的瓶颈是什么？
- 用户反馈最多的问题是什么？
- 哪个方向的投入产出比最高？

基于这些决定下一步做什么，而不是按预设路线图走。

---

## 执行纪律

### 每个Phase的执行流程

```
1. 明确目标（解决什么问题）
2. 验证假设（这个方向真的能解决吗）
3. 写代码（严格按方案）
4. 单元测试（L1）
5. 回归测试（L2——所有历史bug场景）
6. 路径切换测试（L3）
7. 真实用户场景模拟（L4）
8. 对照checklist打勾
9. 全局一致性检查
10. 哲学三问校验
11. 全部通过 → 提交
12. 有任何一项未通过 → 分析原因 → 修复 → 回到步骤4
```

### 禁止事项

- ❌ 不验证假设就写代码
- ❌ checklist没全部打勾就宣布完成
- ❌ 得分下降时调参数而非找根因
- ❌ 用临时补丁代替根本修复
- ❌ 预设六个Phase全部实现——每个Phase完成后重新评估方向

# Phase 1 具体方案（基于Phase 0数据）

## 要做的（有数据支撑）

### 1. 路由修复——给模型一句话辅助提示
假设1显示弱模型不能识别"代码修改"场景。不需要ContextAnalyzer全家桶，只需要：
- 删掉_is_text_task硬编码路由
- 在_decide的判断prompt里加一句：如果conversation history里有代码产出，且用户请求看起来是修改/优化/调整，回复NEED_CODE
- 让模型自己判断，框架只提供提示

### 2. 默认激发语切换到变体1
假设4显示变体1（资深从业者）得分8.04 vs 当前默认变体0得分7.83。直接切换。

### 3. 信号检测修复4个误判case
假设3发现的误判：
- "上面提及的论文"→应该topic_switch但判为engaged（因为有"上面"这个追问词）
- "改成暗色主题"→应该engaged但判为topic_switch（没有追问关键词）
- "嗯，那如果考虑通货膨胀呢"→应该engaged但判为topic_switch
- "好的明白了，另外问下"→应该topic_switch但判为grateful（因为有"好的"）

### 4. 文本聊天保存到历史+response字段
这个不依赖任何假设，直接修。

## 不做的（数据不支持）

### 知识注入（假设2结论：无效）
当前知识注入策略效果-0.08（噪声范围），不投入更多。
暂停_extract_and_store_knowledge，直到找到更好的策略。
