# 连接万物体系评审

# Educe Connector 架构评估

先给结论：**你现在完成的是"连接通路"，还不是"连接万物"。** 通路验证通过是关键里程碑，但从"能跑通"到"可交付特性"还差三个量级的东西——不是代码量，是产品语义。下面逐个拆。

---

## 一、完成度评估

我用一个粗略的分层来看你之前的架构设计意图：

| 层 | 内容 | 状态 |
|---|---|---|
| **传输层** | stdio MCP / JSON-RPC | ✅ 跑通 |
| **抽象层** | Connector ABC / Registry | ✅ 完成 |
| **发现层** | capabilities 发现、两级描述 | 🟡 机制有，效果存疑（见问题2/3）|
| **路由层** | connector.capability 解析、自动查找 | ✅ 基本可用 |
| **生命周期层** | health / shutdown / 重连 / 超时 | 🔴 接口有，逻辑空 |
| **安全层** | 权限、确认、沙箱、审计 | 🔴 完全没有 |
| **可观测层** | 调用日志、失败处理、用户可见状态 | 🔴 缺失 |
| **管理层** | 用户如何增删改 connector | 🔴 手动编辑 json |

**机制完成度 ~70%，产品完成度 ~25%。**

最关键的缺口不是功能，而是这三个：
1. **失败与生命周期** — MCP 子进程崩了怎么办？invoke 超时怎么办？现在大概率是 hang 或抛裸异常。
2. **安全语义** — `filesystem.write_file` 让模型随便调？这是危险动作，目前零防护。
3. **可观测性** — 用户根本不知道"模型刚刚读了你磁盘上哪些文件"。

这三个不补，"连接万物"就是个 demo，不是 feature。

## 二、两级描述的实际效果（问题2）

> 用户只说"搜索 ActivationEngine"，模型走了内置 read_dir 而非 filesystem.search_files

**这是问题，但根因不在 MCP，在你的决策树设计。**

核心区分一个概念：**能力重叠（capability overlap）**。当内置 action 和 connector capability 功能重叠时，你需要一个明确的优先级策略。现在的"决策树优先匹配"是隐式的、不可控的。

我的判断：
- **结果对了 = 好事**。内置 read_dir 能搜到，用户得到答案，这没问题。
- **机制错了 = 隐患**。你不是"选择了更好的工具"，而是"碰巧决策树先匹配"。一旦内置工具不够（比如 search_files 支持正则、内置不支持），模型仍会用内置然后给个烂结果。

**建议：把能力路由从"匹配顺序"改为"能力声明 + 显式优先级"。**

```python
class Capability:
    name: str
    tags: set[str]        # {"file.search", "file.read"}
    preference: int       # 内置默认更高，但可被覆盖
    cost_hint: str        # "local" | "subprocess" | "network"
```

让模型在 system prompt 里看到的不是"两套割裂的工具列表"，而是"按意图组织的能力"。弱模型对"我要搜文件 → 这里有两个工具"的处理能力很差，你得替它收敛。

**更实际的短期方案**：在 system prompt 里加一句能力归并提示，告诉模型何时优先 connector。但这是补丁，不是解法。

## 三、Level 2 按需注入（问题3）

> 弱模型很少主动用 lookup_tools

**这个设计对弱模型基本失效，需要改。** 原因很简单：lookup_tools 要求模型有"我现在信息不足，需要先查询再决策"的元认知，弱模型没有。它要么瞎调，要么放弃。

两级描述本来是为了省 token，但代价是给模型加了一道认知门槛。重新权衡：

**方案 A（推荐）：预算驱动的自适应注入**
- 如果 connector 总 capability 数 < N（比如 30），直接全量注入 Level 2，别搞按需。
- 只有超预算时才退化为 Level 1 + lookup_tools。
- filesystem 才 14 个，根本不需要两级——你现在是用了一个为"上百工具"设计的机制去管 14 个工具。

**方案 B：意图触发的自动 Level 2**
- 不靠模型主动 lookup，而是 orchestrator 在检测到用户意图涉及某 connector 时，**自动**把对应 Level 2 注入下一轮。
- 比如用户提到"文件/搜索/读取" → 自动展开 filesystem 的详细能力。
- 这把"模型主动查询"变成"系统主动喂"，对弱模型友好得多。

**lookup_tools 应该保留，但定位为强模型的 affordance，不能作为弱模型的依赖路径。** 你的实测已经证明了这点。

## 四、"万物"到底是哪些（问题4）

先纠正一个潜在误区：**"连接万物"不等于"内置一堆 connector"。** 如果你自己写 Git connector、DB connector、Slack connector……你就变成了又一个 LangChain，永远追不上生态。

正确的姿势是：

**1. MCP 生态就是你的"万物"。** filesystem、git、postgres、github、puppeteer、slack… MCP 官方和社区已经有几十上百个 server。你的价值是**接入层**，不是 server 本身。

**2. 你需要补的不是 connector，是 transport 和管理能力：**
- ✅ stdio（已做）
- 🔴 **SSE / HTTP transport** — 远程 MCP server，这是企业场景刚需
- 🔴 **OAuth / 凭据管理** — 连 GitHub、Slack 这类需要授权的服务

**3. 你自己只该写两三个"战略 connector"：**
- **BuiltinConnector**（已做）— 框架自身能力
- **可能的 Educe 内部 connector** — 比如把"激发引擎/seed"本身暴露成 connector，让 Agent 能操作自己的演化系统（这才是 Educe 的独特性，见下一节）

**结论：判定"万物体系完成"的标准不是 connector 数量，而是：任意一个标准 MCP server，用户能在 5 分钟内接入并安全使用。** 现在你做到了"开发者手动配 json 能接入 stdio server"，距离这个标准还差:transport 多样性 + 凭据 + 管理 UI + 安全。

## 五、与激发引擎的结合（问题5）

**这是你这个问题里最有价值的一条，也是 Educe 区别于其它 Agent 框架的护城河所在。** 别把它当锦上添花。

其它框架的工具调用是无状态的、用完即忘。Educe 有 seed 演化，意味着**工具使用应该成为演化信号**。几个交叉点：

**1. Connector 使用效果 → seed 演化（你提到的方向，对的）**
- 记录每次 invoke 的：是否成功、用户是否接受结果、是否引发追问。
- 把"哪些 connector / capability 在哪些情境下有效"沉淀进 seed。
- 比如：某 seed 演化出"处理代码搜索任务时，search_files 比 read_dir 更有效"的偏好。这让能力路由（问题2）从静态优先级变成**演化出来的策略**。

**2. Seed 反向影响 capability 暴露**
- 不是所有 connector 对所有 seed 都该可见。一个专注"文档写作"的 seed 不需要 postgres connector 占 token。
- 让 seed 携带"能力亲和性"，动态裁剪注入的 capability 集合。这同时解决了问题3的 token 预算。

**3. Connector 作为 seed 的"感官"和"手"**
- 概念上：seed 是 Agent 的"性格/策略演化体"，connector 是它"感知和作用于世界"的接口。
- 一个成熟的 seed 应该"知道自己能用哪些 connector，擅长用哪些"。

**建议落地顺序**：先做 (1) 的最小版——只记录 invoke 的成功/失败/采纳，写进 seed 的经验。不要一上来做复杂的演化策略。**先有数据闭环，再谈演化。**

## 六、产品视角：用户如何"连接"（问题6）

当前流程是手动编辑 `.deepforge/mcp.json`。**对开发者够用，对产品不够。**

但别急着做 UI。先想清楚分层用户：

| 用户类型 | 接受的连接方式 |
|---|---|
| 框架开发者（你）| 手编 json ✅ 够了 |
| 高级用户 | json + 文档 + 验证反馈 |
| 普通用户 | 一键安装 / 市场 / 自然语言"帮我连上 GitHub" |

**短期（可交付门槛）至少要补：**
1. **配置验证 + 错误反馈** — json 写错了，用户得到的不能是堆栈跟踪。要有"filesystem connector 启动失败：找不到 npx"这种人话。
2. **连接状态可见** — 在 UI 里显示哪些 connector 在线/离线/能力数。health() 你有接口了，把它接到前端。
3. **首次连接的能力预览** — 连上后告诉用户"已连接 filesystem，新增 14 项能力如读取文件、搜索文件"。

**中期（产品化）：**
- Connector 市场 / 模板（从已知 MCP server 列表一键添加）
- 凭据安全存储

## 七、"连接万物"还差几步才能交付

我给你一个明确的交付清单，按优先级：

**P0 — 不补不能交付（这是底线，缺了就是危险 demo）**
1. **Invoke 超时 + 错误兜底** — 不能 hang，不能裸异常给用户
2. **MCP 子进程生命周期** — 崩溃检测、shutdown 清理、避免僵尸进程
3. **危险能力的确认机制** — write_file 这类至少要能配置"需用户确认"
4. **调用可观测** — 用户能看到"模型调用了什么、参数、结果"

**P1 — 决定特性好不好用**
5. 能力路由收敛（问题2的能力声明方案）
6. Level 2 自适应注入（问题3的方案A或B，先用A，filesystem 直接全量）
7. 配置错误的人话反馈 + 连接状态 UI

**P2 — 决定特性能不能规模化**
8. SSE/HTTP transport
9. 凭据管理
10. Connector 市场/模板

**P3 — Educe 独特价值**
11. Connector 使用 → seed 演化的数据闭环（先记录，后演化）

## 总结判断

- **你跑通了最难的技术验证**（MCP 通路 + 弱模型能调用），这步含金量很高，恭喜。
- **但"连接万物"作为可交付特性，现在大约在 30%。** 缺的不是更多 connector，是 P0 的安全/生命周期/可观测——这三个是"玩具 vs 产品"的分界线。
- **不要自己写一堆 connector**,把力气花在接入层质量和 MCP 生态对接上。
- **最具 Educe 特色、也最该尽早埋点的是 connector × seed 闭
