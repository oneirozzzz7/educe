# Opus 4.8 — 模型不反问问题讨论

# 反问 vs 执行：人机权责边界的设计问题

## 先纠正一个归因偏差

你的问题分析里，1 和 2 是并列的——但实际权重严重不对等。

**这大概率是 prompt 的问题，不是模型的问题。**

判断依据：Qwen3.6 这一代指令遵循模型，对"反问"是有能力的。你给它一句"遇到歧义先确认"，它就会问。它现在不问，是因为你的 prompt 里有"直接执行不要解释"这种**强指令**在压制它。

模型不会区分"不要解释你的代码"和"不要确认需求"——在它眼里这都是"少废话、动手干"。你写的是针对话痨问题的解药，副作用是把反问能力一起阉割了。

所以先做一个实验再下结论：

> 临时把"直接执行不要解释"删掉，换成一句中性的"必要时可以提问"，跑那几个模糊需求，看模型会不会问。

如果它开始问了 → 是 prompt 问题（90% 概率）。
如果还是不问 → 才需要考虑模型倾向 + 更强的引导。

**不要在没做这个实验前就归因到"模型训练倾向"。** 这是甩锅最舒服但最没用的结论。

---

## 为什么 Claude Code 会问、你的不会

不是因为模型更聪明。是因为行为被显式定义了。

Claude Code 的系统提示里有大量关于"何时澄清"的规则，而且关键在于：**它把"问"和"执行"放在了一个明确的决策框架里，而不是一句模糊的"必要时确认"。**

你现在的 prompt 给的是单极指令（"执行"），自然得到单极行为。

---

## 核心：反问 vs 执行的决策逻辑

这是设计的真正难点。"遇到歧义就问"是错的——会变成另一个极端：什么都问，变成话痨助手，体验更差。

真正的判断维度是 **歧义的代价**，不是歧义的有无。

### 决策矩阵

```
                  错误成本低          错误成本高
              （改回来很容易）    （改回来很贵/不可逆）
            ┌──────────────────┬──────────────────┐
  歧义大    │  做合理假设+声明   │   必须反问        │
（多种合理解）│  "我假设X，先做"  │                  │
            ├──────────────────┼──────────────────┤
  歧义小    │   直接执行         │  直接执行+声明假设 │
（意图清晰） │                  │                  │
            └──────────────────┴──────────────────┘
```

**两个轴：**

1. **歧义度**：是否存在多种合理解释，且它们导向显著不同的产出？
2. **错误成本**：如果假设错了，纠正的代价多大？

### 套用你的例子

| 场景 | 歧义 | 错误成本 | 决策 |
|------|------|---------|------|
| "帮我做个待办管理器" | 高（CLI/Web/存储都不定） | 高（架构选错要重写） | **反问** |
| "改一下这个文件" | 高（改哪、改成啥都不知道） | 中-高 | **反问** |
| "把这个函数的变量名改成驼峰" | 低 | 低 | **直接执行** |
| "加个错误处理" | 中（怎么处理法） | 低（容易调整） | **执行+声明假设** |

关键洞察：**"做合理假设并声明"是第三条路**，比"纯执行"和"纯反问"都重要。它让模型在低成本场景下保持高效，同时给用户一个纠错锚点。

## Prompt 设计建议

不要写"必要时反问"——太模糊。给出可操作的触发条件：

```
## 行动前的判断

在执行任务前，评估两件事：

1. 是否存在多种合理理解，且它们会导向明显不同的产出？
2. 如果我的理解错了，用户纠正的代价有多大？

据此选择：

- 意图明确 → 直接执行
- 有歧义但选错了容易改 → 选最合理的方案执行，
  并用一句话声明关键假设（"我按 X 来做，需要的话告诉我"）
- 有歧义且选错了代价大（架构选型、删除数据、大范围改动、
  不可逆操作） → 先提问澄清，不要动手

## 提问的方式

需要澄清时：
- 一次问完关键问题，不要挤牙膏
- 给出带默认选项的封闭式问题，降低用户回答成本
  ✓ "用 CLI 还是 Web？如果没偏好我默认做 CLI"
  ✗ "你想要什么样的待办管理器呢？"
- 不确定的点 ≤3 个，超过说明任务本身需要拆解
```

注意最后这条"带默认选项"——这是 Claude Code 做得好的地方：**问的同时给一个倾向，用户可以一句"按你说的来"就过关。** 这避免了反问变成负担。

## 从治理层看：这是权责边界，不只是体验

你点到了关键。换个语言说清楚：

**反问行为的本质，是模型在"承担决策责任"和"把责任交还给人"之间做选择。**

- 模型擅自假设并执行 → 它在替用户做了本该用户做的决策，**越权**
- 模型什么都问 → 它把本该自己承担的判断推给用户，**失职**

好的边界是：

> 模型自主决策的范围，应当与"决策出错后的可恢复性"成正比。

可逆、低成本的决策，模型应该自己扛（不要烦用户）。
不可逆、高成本的决策，决策权必须回到人手里（不能替用户冒险）。

这条原则可以直接写进你的治理层规范，作为所有 agent 行为的元规则——不止用于反问，也用于"是否要二次确认删除""是否要在执行破坏性操作前停下"等。

## 落地优先级

1. **先做归因实验**（删掉压制性指令，验证是 prompt 问题）——半天
2. **引入决策矩阵到 prompt**，用上面的措辞——1 天
3. **用真实模糊需求做评测**，看模型分类是否准确——记录它在哪些 case 上误判
4. 把"可恢复性 ∝ 自主权"写进治理层元规范
