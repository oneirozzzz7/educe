# Educe Plan/Build 不稳定问题：架构师分析

先说结论：你的假设方向对了，但没触及最深的根因。让我重新拆。

## 1. 根因分析：认知层面的真问题

你的 read_dir / read_file / memorize 之所以稳定，**不是因为映射简单，而是因为它们的 action 是"动作的终点"——做完就有结果**。模型脑子里有一个清晰的"输入→工具→输出"闭环。

而 plan 和 build 失败的真正原因，是它们触发了弱模型的**两种认知冲突**：

### 冲突 A：plan 是"元动作"，违反了模型的语言直觉

read_file 是"做一件事"。
plan 是"描述将要做的事"。

对弱模型来说，**"描述步骤"和"用自然语言说步骤"在它的训练分布里是同一件事**。它没有"我说出步骤是为了交给一个执行器去跑"这个概念。在它见过的语料里，"规划"99% 的情况下就是用文字回答。

你让它把规划塞进 `<action type="plan">`，本质是要求它做一个**反直觉的格式化包装**：把"已经是答案的东西"再包一层。弱模型的指令遵循能力不足以压制这个强先验。

**关键洞察：plan 不该是一个 action。它是模型的"思考结果"，不是"动作"。** 你把思考伪装成了动作，弱模型识破了这个伪装。

### 冲突 B：build 和 plan 共享同一个语义触发器

"帮我写个应用" 这句话同时激活了：
- "我需要规划" (plan)
- "我需要构建" (build)

弱模型缺乏的不是知识，是**优先级仲裁能力**。当两个 action 的触发条件重叠时，它会随机坍缩，或者退回最熟悉的模式（用文字描述）。

你的 prompt 里 build 和 plan 是**并列罗列**的，没有任何"谁先谁后/谁包含谁"的结构。弱模型看到一堆并列选项，又遇到一个模糊请求，仲裁失败。

### 更深的根因：你在用"动作列表"喂模型，但弱模型需要"决策树"

强模型能从扁平列表里自己构建决策逻辑。弱模型不能。它需要**被牵着走的状态机**。

## 2. 大胆的解决方案

### 方案一：消灭 plan action —— 让 plan 变成"框架行为"而非"模型行为"

**核心思想：plan 不是模型要主动产出的格式，而是框架在检测到模型"想多步做事"时自动接管的状态。**

具体做法：取消 `<action type="plan">`。改成模型只会一种东西——**连续产出单步 action**。

当框架检测到模型在一轮里输出了"我打算先 A 再 B 再 C"这种自然语言意图时，框架反问一个**结构化确认**：

```
框架: 我理解你要分3步完成。要我逐步自动执行吗？
[模型只需回答 yes/no 或直接给出第一个 action]
```

也就是说，**让模型干它最擅长的事（用自然语言描述步骤），框架负责把它结构化**。

- **预期效果**：plan 触发率从 1/9 直接变成 N/A——因为不再需要模型产出 plan 标签。弱模型用自然语言描述步骤是它的强项（3.6-35B 这点没问题）。
- **权衡**：违反了你"不硬编码流程"的哲学吗？不完全。框架没有硬编码"该怎么规划"，只是把"自然语言意图→可执行计划"这个**翻译**承包了。这是放大，不是限制。
- **对强模型**：强模型可以继续直接产出结构化计划，框架识别后跳过反问。无损。

### 方案二：单一入口 + 渐进收敛（解决 build/plan 混淆）

**核心思想：弱模型仲裁不了并列选项，那就只给它一个入口。**

把 build 和 plan 合并成一个动作 `<action type="work">`：

```
<action type="work">用户的原始需求</action>
```

模型遇到任何"帮我做X"的请求，**只需要识别"这是一个要干活的请求"**（这个判断弱模型做得到，9/9 级别），然后无脑包进 work。

接下来由框架决定：这个 work 需要规划吗？需要的话，框架内部调用一次 LLM（可以是同一个模型）做"该拆步骤吗"的单点判断——**把复杂的二选一变成两次简单的单点判断**。

- **预期效果**：build 混淆问题消失，因为模型不再需要在 build/plan 间选择。
- **权衡**：增加一次 LLM 调用（成本/延迟）。但对弱模型，**用 token 换稳定性是值得的**。
- **认知原理**：把"分类问题"降级成"识别问题"。弱模型的识别能力 >> 分类能力。

### 方案三：行为表达从"列表"改成"二叉决策树"（结构化 prompt）

不改架构，只改 prompt 的**信息结构**。弱模型需要决策树，那就给它画一棵：

```
## 你的决策流程（严格按顺序判断）

第一步：用户是想"了解情况"还是"改变什么"？
  ├─ 了解情况 → 用 read_dir / read_file / recall（直接执行，无需确认）
  └─ 改变什么 → 进入第二步

第二步：是一句话能做完，还是需要多步？
  ├─ 一句话做完 → 用 shell / write_file（单个 action）
  └─ 需要多步 → 用 build，把完整需求原样交给框架，由框架拆步骤
                ⚠️ 注意：你不需要自己写步骤！只需把需求交给 build。
                ⚠️ "帮我写个应用"= build，不是 plan。
```

并且用**反例锚定**（弱模型对反例敏感）：

```
❌ 错误：用户说"帮我写个待办应用"，你回答"步骤：1.创建项目 2.写界面..."
✅ 正确：<action type="build">写一个待办应用</action>
```

- **预期效果**：成本最低，能立刻提升。预估 plan 触发率提升到 5-6/9。
- **权衡**：治标不治本。决策树越长弱模型越容易在深层节点失守。但作为兜底很好。

### 方案四（最大胆）：Few-shot 轨迹注入 + 动作语法约束（约束解码）

**双管齐下：**

**A. 在每次推理时，根据用户输入的语义，动态注入一条最相似的"成功轨迹"作为 few-shot。** 弱模型是模仿器不是推理器，给它一个几乎一样的范例，模仿成功率极高。维护一个小的"黄金轨迹库"（你的 9 个 case 本身就是种子）。

**B. 用约束解码 (constrained decoding) 强制语法。** 既然你不用分类器，那就在**生成阶段**约束：当模型开始生成 `<action type="` 时，用 grammar 限制后续只能是合法 type，并且对于"明显是干活请求"的上下文，用 logit bias 提升 `build` 的概率。

- **预期效果**：这是最强组合，弱模型触发率可逼近强模型。
- **权衡**：需要你能控制推理栈（vLLM/SGLang 的 grammar/logit bias）。如果你只能调 API，B 不可行。轨迹库需要维护。

## 3. 我最推荐的方案

**方案一 + 方案二的组合，外加方案三作为 prompt 兜底。**

理由：

1. **plan 的问题是设计错误，不是 prompt 不够好。** plan 作为一个要求模型主动产出结构化格式的 action，本身就在和弱模型的语言先验对抗。最优雅的解法是**取消这个对抗**——让模型用自然语言表达意图，框架来结构化。这不是妥协，这是更符合"框架放大能力"哲学的设计：框架补上了弱模型缺的那块（结构化能力）。

2. **build/plan 混淆的根源是给了模型一个它做不了的仲裁。** 合并成单一 `work` 入口，把仲裁从模型转移到框架，是最干净的解。

3. **这套设计对强模型零损害甚至更好**：强模型直接产出结构化 work + 步骤，框架识别后直接执行；弱模型产出自然语言，框架兜底。**同一套接口，自动适配模型能力**——这才是真正的"框架激发弱模型潜力"。

核心哲学升级：

> **不要让模型适应框架的格式，让框架适应模型的表达。**
> action 标签应该是模型能力的下限保障，而不是上限要求。

## 4. 验证实验设计

### 实验目标
验证"取消 plan action + 合并 work 入口"是否显著提升弱模型稳定性，且不损害强模型。

### 变量
- 自变量：prompt/架构方案（4 个臂）
  - **A0** 现状（baseline）
  - **A1** 方案三（决策树 prompt，纯 prompt 改动）
  - **A2** 方案一+二（架构改动：work 入口 + 框架接管 plan）
  - **A3** A2 + 方案四A（轨迹注入）
- 控制变量：模型（弱模型 Qwen3.6-35B-A3B + 至少一个强模型如 Claude/GPT-4 做对照）

### 数据集
现有 9 个 case 太少，无法区分信号和噪声。**扩展到每类 30 个 case**，重点扩充：
- plan 类：20 个不同表述（"规划一下""帮我分步""怎么一步步做"...）
- build 类：20 个（"帮我写""做一个""实现一个"...）
- **对抗类（关键）：15 个 build/plan 边界模糊的 case**（如"帮我设计并实现一个X"——故意同时触发两个语义），专门测仲裁。
- 干扰类：10 个根本不该触发任何 action 的闲聊，测假阳性。

每 case 跑 5 次（不是 3 次，3 次方差太大）。

### 核心指标
1. **首动作正确率**：第一个 action 的 type 是否正确（主指标）
2. **任务完成率**：整个轨迹最终是否达成目标（终极指标）
3. **格式合规率**：是否产出合法 action 标签
4. **假阳性率**：不该触发 action 时是否乱触发（防止过度修复）
5. **强模型回归**：强模型在 A2/A3 上是否 ≥ A0（守门指标，不能退化）

### 关键设计：分离"格式问题"和"决策问题"
你现在的 1/9 混在一起，分不清模型是"不会选 plan"还是"选对了但没产出标签"。实验里要**单独标注每次失败的归因**：
- 类型 1：选错了 action（决策失败）
- 类型 2：选对了但用自然语言（格式失败）
- 类型 3：根本没动作（理解失败）

这个归因表会告诉你方案该往哪个方向加力。

### 判定标准
- 主要：A2 在弱模型上首动作正确率 ≥ 85%（对标 read_dir 的稳定性）
- 守门：A2 在强模型上不低于 A0
- 若 A2 已达标，A3 的边际收益要权衡轨迹库维护成本
